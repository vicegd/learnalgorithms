# Asymptotic Notation & Big-O

In the previous section, we calculated the exact polynomial execution time of a function, such as $T(n) = 3n + 4$. While mathematically precise, dealing with exact polynomials becomes impossible as software architectures grow to millions of lines of code.

To evaluate large-scale systems, engineers use **Asymptotic Notation**. This mathematical tool allows us to describe the *rate of growth* of an algorithm by isolating its dominant term and ignoring hardware-specific constants.

---

## 1. The Formal Definition of Big-O ($O$)

The Big-O notation, or Omicron, establishes a mathematical **upper bound** on the growth of a function. It represents the worst-case scenario scaling.

**Formal Definition:** A function $T(n)$ is said to be $O(f(n))$ if and only if there exist two positive real constants, $c$ and $n_0$, such that:

$$T(n) \le c \cdot f(n), \quad \forall n \ge n_0$$

### Mathematical Proof Example
Let's prove that the polynomial $T(n) = 5n^4 + 4n^3$ belongs to the complexity class $O(n^4)$.
We need to find a constant multiplier $c$ that makes $c \cdot n^4$ strictly greater than our polynomial for any input larger than $n_0$.

If we assume $n \ge 5$ (so, $n_0 = 5$), we can state:
$$5n^4 + 4n^3 \le 5n^4 + n^4$$
$$5n^4 + 4n^3 \le 6n^4$$

By selecting $c = 6$ and $n_0 = 5$, the inequality holds true to infinity. Therefore, the lower-degree terms and constants are absorbed, proving the algorithm is strictly **$O(n^4)$**.

---

## 2. Algebraic Properties of Complexity

When analyzing complex code architectures, we don't calculate limits manually every time. We apply standard simplification rules:

* **Identity:** $O(T(n)) = T(n)$
* **Constant Drop:** $O(c \cdot f(n)) = O(f(n))$. Hardware speed ($c$) does not change the asymptotic class.
* **The Sum Rule (Sequential Code):** If a program executes one block of code followed by another, the total complexity is determined by the dominant (slowest) block:
  $$O(f(n)) = \max(O(T_1(n)), O(T_2(n)))$$
* **The Product Rule (Nested Code):** If an algorithm executes a loop *inside* another loop, their individual complexities multiply:
  $$O(f(n)) = O(T_1(n) \cdot T_2(n))$$

## 3. The Logarithmic Baseline

Logarithms are the backbone of optimal computer science (especially in algorithms that divide search spaces in half, like Binary Search). However, algebraic misunderstandings here often lead to fatal errors in algorithm design. 

:::info Base Equivalence
In asymptotic notation, the base of the logarithm is irrelevant due to the mathematical base-change property, which only introduces a constant multiplier. 
Therefore: $O(\log_2 n) = O(\log_{10} n) = O(\log n)$.
:::

Test your mathematical foundation with these critical properties before proceeding:

<details>
  <summary>🧠 <strong>Math Check 1:</strong> Is $n^{\log_3 a} = a^{\log_3 n}$?</summary>
  <strong>YES.</strong> This is due to the fundamental logarithm base-interchange property ($n^{\log_a b} = b^{\log_a n}$). This property is heavily used in the Master Theorem for Divide and Conquer algorithms.
</details>

<details>
  <summary>🧠 <strong>Math Check 2:</strong> Is $O(\log_3 n)^8 = O(8 \log_3 n)$?</summary>
  <strong>NO.</strong> A massive difference exists. The first is a polylogarithmic function of degree 8 (the result of the logarithm is raised to the 8th power). The second applies the power rule $O(\log n^k) = O(k \log n)$, which simply drops the constant $8$, making it a linear logarithmic function. 
  <br/><strong>Rule:</strong> $O(\log^k n) \neq O(\log n)$.
</details>

<details>
  <summary>🧠 <strong>Math Check 3:</strong> Is $\log_3(4n) = \log_3 4 \cdot \log_3 n$?</summary>
  <strong>NO.</strong> The product property of logarithms transforms multiplication into addition, not another multiplication. The correct expansion is: $\log_3(4n) = \log_3 4 + \log_3 n$.
</details>

## 4. The Hierarchy of Complexities

When designing a system, your goal is to push the algorithm as high up this list as mathematically possible:

1. **$O(1)$ (Constant):** The absolute ideal. Execution time is independent of the input size.
2. **$O(\log n)$ (Logarithmic):** Excellent. The algorithm discards fractions of the dataset repeatedly (e.g., Binary Search).
3. **$O(n)$ (Linear):** Very good. The algorithm scans the data optimally without nested loops.
4. **$O(n \log n)$ (Linearithmic):** The physical limit for comparison-based sorting algorithms (e.g., MergeSort).
5. **$O(n^2)$ (Quadratic):** Standard for double-nested loops. Acceptable only for very small datasets.
6. **$O(n^k)$ (Polynomial):** Highly inefficient as $k \ge 3$, but technically computable.
7. **$O(2^n)$ (Exponential):** The danger zone. Execution time doubles with every single item added.
8. **$O(n!)$ (Factorial):** Completely unmanageable. Finding every permutation of a deck of cards ($52!$) would take longer than the current lifespan of the universe.

## 5. The Hardware Reality (Scaling Tables)

*“Why don't we just buy a faster supercomputer in the cloud instead of redesigning the $O(n^3)$ algorithm?”*

To answer this eternal management question, look at how different complexity classes react to hardware scaling.

<Tabs>
  <TabItem value="time" label="⏱️ Execution Time Scaling" default>
    If an algorithm processes an input size of $n_1 = 1,000$ in exactly **1 unit of time (u.t.)**, how long will it take if we scale the company's data to one million records ($n_2 = 10^6$)?

    | Complexity Class | Time for $n = 10,000$ | Time for $n = 1,000,000$ |
    | :--- | :--- | :--- |
    | **$O(n)$** | $10 \text{ u.t.}$ | $1,000 \text{ u.t.}$ |
    | **$O(n^2)$** | $100 \text{ u.t.}$ | $1,000,000 \text{ u.t.}$ |
    | **$O(2^n)$** | $2^{9000} \text{ u.t.}$ | The heat death of the universe. |

  </TabItem>
  <TabItem value="size" label="💾 Maximum Problem Size">
    Conversely, if our current hardware can process an input of size $n_1 = 10$ in 1 unit of time, what happens to the maximum problem size we can handle if we buy a computer **one million times faster**?

    | Complexity Class | Max Size ($1,000\times$ faster CPU) | Max Size ($1,000,000\times$ faster CPU) |
    | :--- | :--- | :--- |
    | **$O(n)$** | $n_2 = 10^4$ | $n_2 = 10^7$ |
    | **$O(n^3)$** | $n_2 = 100$ | $n_2 = 1,000$ |
    | **$O(2^n)$** | $n_2 \approx 20$ | $n_2 \approx 30$ |

    :::danger The Principle of Invariance
    A million-dollar hardware upgrade only increases the capacity of an exponential algorithm by **20 extra items**. Asymptotic complexity defeats hardware. Always.
    :::
  </TabItem>
</Tabs>
