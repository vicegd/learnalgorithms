# The Backtracking Paradigm

Backtracking is an algorithm design paradigm used to solve problems where a solution must be constructed by making a sequence of decisions under constraints. Instead of generating every possible complete candidate and validating it at the end, backtracking builds solutions incrementally and abandons partial solutions as soon as they can no longer lead to a valid answer.

This idea makes backtracking one of the most important techniques for solving combinatorial search problems.

Typical applications include:

- N-Queens
- Sudoku
- Subset Sum
- Graph Coloring
- Hamiltonian Cycle
- SAT and Constraint Satisfaction Problems (CSPs)
- Timetabling and scheduling problems

---

# Why Brute Force Is Not Enough

Many combinatorial problems suffer from **combinatorial explosion**.

Consider the Magic Square problem.

- 3×3: 9! = 362,880 arrangements
- 5×5: 25! ≈ 1.55 × 10^25 arrangements

A brute-force algorithm generates every possibility.

Backtracking attempts to detect failure early and avoid exploring large portions of the search space.

---

# State-Space Search

Backtracking is usually viewed as a **state-space search** technique.

A problem is represented as a collection of states:

- Nodes represent states.
- Edges represent decisions.
- Leaves represent completed solutions or dead ends.

The search space is normally explored using Depth-First Search (DFS).

## Explicit vs Implicit Graphs

In most realistic problems the state-space graph is too large to store explicitly.

Instead, backtracking uses an implicit graph:

- States are generated on demand.
- Only the current path is kept in memory.
- Previously explored branches can be discarded.

This significantly reduces memory consumption.

---

# Search Tree and Decision Levels

A solution can be expressed as:

$$
(x_1,x_2,\ldots,x_n)
$$

A partial solution is:

$$
(x_1,x_2,\ldots,x_k)
$$

with:

$$
k<n
$$

Each level of the state tree corresponds to one decision variable.

```text
Level 0 : []
Level 1 : [x1]
Level 2 : [x1,x2]
...
Level n : Complete solution
```

---

# Formal Model

Many backtracking problems can be formalized as:

$$
(X,D,C)
$$

where:

- X = variables
- D = domains
- C = constraints

The objective is to find an assignment satisfying every constraint.

This formulation is identical to a Constraint Satisfaction Problem (CSP).

# Constraint Satisfaction Problems (CSPs)

A CSP contains:

1. Variables
2. Domains
3. Constraints

Examples:

- Sudoku
- N-Queens
- Graph Coloring
- Course Scheduling
- Resource Allocation

Backtracking is one of the most widely used generic techniques for solving CSPs.

# The Choose-Explore-Undo Pattern

Every backtracking algorithm follows the same fundamental idea:

1. Choose a decision.
2. Explore the consequences.
3. Undo the decision.

The undo step is what gives the paradigm its name.

```text
Choose
  ↓
Explore
  ↓
Failure?
  ↓
Undo
```

---

# Generic Backtracking Pseudocode

```text
BACKTRACK(state)

    if isSolution(state) then
        process(state)
        return

    for each choice do

        apply(choice)

        if promising(state) then
            BACKTRACK(state)

        undo(choice)
```

This language-independent template captures the essence of the paradigm.

---

# Promising States

A state is called promising if it can still lead to a valid solution.

```text
Generate state
       │
       ▼
Promising?
       │
 ┌─────┴─────┐
 │           │
 No         Yes
 │           │
 ▼           ▼
Prune    Continue
```

The quality of the promising function often determines the practical efficiency of the algorithm.

# DFS and Backtracking

Backtracking is frequently described as DFS with pruning.

DFS determines:

- Exploration order.

Backtracking adds:

- Constraint checking.
- Pruning.
- Rollback.
- Solution construction.

Therefore:

```text
DFS ≠ Backtracking
```

but most backtracking algorithms naturally use DFS.

---

# Worked Example: Subset Sum

Given:

```text
Numbers = {3,5,6,7}
Target = 12
```

At every level we decide whether to include a number.

```text
                   {}
              /          \
            {3}          {}
          /    \
      {3,5}   {3}
```

Suppose we reach:

```text
{3,5,6}
```

Current sum:

$$
3+5+6=14
$$

Since:

$$
14 > 12
$$

and all remaining numbers are positive, the branch can never produce a valid solution.

```text
PRUNE
```

A brute-force algorithm would still explore all descendants.

# Example: N-Queens

The goal is to place n queens on an n×n chessboard.

Constraints:

- No shared row.
- No shared column.
- No shared diagonal.

For n = 4:

```text
. Q . .
. . . Q
Q . . .
. . Q .
```

Whenever a new queen attacks a previous queen:

```text
Invalid state
    ↓
Prune
```

This dramatically reduces the search space.

# Java Blueprint: First Solution

```java
boolean backtrack(State state) {

    if (isSolution(state)) {
        System.out.println(state);
        return true;
    }

    for (State child : state.getChildren()) {

        if (backtrack(child)) {
            return true;
        }
    }

    return false;
}
```

---

# Java Blueprint: All Solutions

```java
void backtrack(State state) {

    if (isSolution(state)) {
        process(state);
        return;
    }

    for (State child : state.getChildren()) {
        backtrack(child);
    }
}
```

# Pruning

Pruning is the key reason backtracking outperforms naive exhaustive search in practice.

## Sources of Pruning

### Constraint Violation

```text
Two queens attack each other.
```

→ Prune

### Domain Exhaustion

```text
No valid values remain.
```

→ Prune

### Resource Limits

```text
Cost > Budget
```

→ Prune

### Bounding Information

```text
Cannot beat current best solution.
```

→ Prune

# Complexity Analysis

Let:

- b = branching factor
- d = maximum depth

Then the search tree contains approximately:

$$
1+b+b^2+...+b^d
$$

Therefore:

$$
T(n)=O(b^d)
$$

## Exponential Growth

If each level offers two choices:

$$
T(n)=O(2^n)
$$

## Factorial Growth

Permutation problems often generate:

$$
O(n!)
$$

states.

Pruning may significantly reduce the number of explored nodes, but worst-case complexity usually remains exponential.

---

# Space Complexity

Backtracking usually stores only:

- The current path.
- Recursive calls.
- Local state.

If the maximum depth is h:

$$
M(h)=O(h)
$$

This is one of the biggest advantages of DFS-based search.

# Backtracking vs Brute Force

| Brute Force | Backtracking |
|------------|-------------|
| Generates complete candidates | Builds incrementally |
| Validates at the end | Validates during construction |
| Explores all branches | Prunes branches |
| Usually slower | Often much faster in practice |

---

# Backtracking vs Dynamic Programming

Backtracking:

- Searches a state tree.
- Useful for constraints and enumeration.

Dynamic Programming:

- Reuses solutions to overlapping subproblems.
- Exploits optimal substructure.

If an efficient DP formulation exists, DP is often preferable.

---

# Backtracking vs Branch and Bound

Backtracking:

```text
Prune because a state is invalid.
```

Branch & Bound:

```text
Prune because a state cannot improve the current optimum.
```

Branch & Bound is often considered an optimization-oriented extension of backtracking.

---

# Design Methodology

When designing a backtracking algorithm, answer:

- What is the state?
- What are the decisions?
- What is a complete solution?
- Which constraints can be checked early?
- Which branches can be pruned?
- Do we need one solution, all solutions, or the best solution?

# Common Implementation Pitfalls

## Forgetting Undo

Incorrect:

```java
solution.add(x);
backtrack();
```

Correct:

```java
solution.add(x);
backtrack();
solution.remove(solution.size()-1);
```

## Checking Constraints Too Late

Bad:

```text
Generate everything.
Validate later.
```

Good:

```text
Validate continuously.
```

## Excessive State Copying

Prefer:

```text
Choose
Explore
Undo
```

instead of cloning large structures at every recursive call.

Backtracking does not change theoretical complexity, but powerful pruning strategies often make practical solutions possible.

# Key Ideas to Remember

- Backtracking explores an implicit state-space tree.
- Solutions are built incrementally.
- DFS is the usual exploration strategy.
- Pruning is the key to efficiency.
- Complexity is frequently exponential.
- Memory usage is usually linear in search depth.
- The paradigm is fundamental for CSPs, combinatorial search, and many NP-complete problems.
